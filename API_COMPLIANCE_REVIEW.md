# Проверка соответствия ТЗ и интеграции Sletat API

Дата проверки: 2026-03-10.

## 1) Что удалось проверить

- В репозитории есть ТЗ (`Тз Бот Горящих Туров Mvp(1).docx`) и реализация NestJS-бота.
- Прямой доступ к указанным страницам wiki.sletat.ru из текущего окружения отсутствует (HTTP 403), поэтому сверка по полям/контрактам выполнена по коду проекта и по требованиям из ТЗ.

## 2) Ключевой вывод

Текущая реализация **не соответствует** требованиям полноценной интеграции с API Sletat (json/xml шлюзы поиска и детальная актуализация), потому что в проекте подключён только MOCK-клиент.

## 3) Подтверждение по коду

1. В модуле Sletat зарегистрирован `SletatMockService` как основной клиент (`SLETAT_CLIENT`).
2. В `SletatMockService` все ключевые методы содержат комментарий `MOCK IMPLEMENTATION` и `TODO: replace with real Sletat API call`.
3. В проекте отсутствует отдельная реализация HTTP-клиента под XML/JSON-шлюзы (поиск туров, детальная актуализация, бронирование/платежи) и маппинг реальных полей ответов поставщика.

## 4) Соответствие ТЗ по функциональности (кратко)

### Реализовано частично

- Telegram-бот принимает текст и голос.
- Есть распознавание голоса через OpenAI Whisper.
- Есть парсинг текста в структурированный запрос.
- Есть выдача вариантов, подписки на мониторинг, базовая anti-spam логика, и сценарий "бронирования".

### Не выполнено / не подтверждено

- Реальная интеграция со шлюзами Sletat (json/xml) и справочниками поставщика.
- Реальная детальная актуализация оффера по документации Sletat.
- Реальный контур заявки/оплаты Sletat (в коде mock URL).
- Полная валидация обязательных параметров перед запуском поиска согласно ТЗ (часть проверок делегирована LLM, но жёсткой серверной валидации критических полей нет).

## 5) Что нужно сделать для соответствия документации Sletat

1. Добавить новый клиент, например `SletatApiService`, реализующий `SletatClient`:
   - поиск (json/xml согласно выбранному шлюзу),
   - детальная актуализация,
   - справочники,
   - создание/проверка заявки,
   - платежные ссылки.
2. Переключить DI-провайдер `SLETAT_CLIENT` с mock на реальный клиент через конфиг (`SLETAT_MODE=mock|api`).
3. Добавить строгий маппинг и валидацию входных/выходных DTO по полям документации.
4. Добавить интеграционные тесты с фикстурами реальных ответов API.
5. Добавить логирование внешних вызовов/ошибок и retry/backoff для нестабильных API-вызовов.

## 6) Итог

С точки зрения ТЗ проект выглядит как MVP-заготовка с правильной архитектурой модулей, но **не как завершённая интеграция с документацией Sletat**: пока реализован только MOCK-слой.
